=== SLIDE 2 ===
North-star product experience
The platform should feel like one coherent studio that guides a user from fuzzy intent to safe deployment and continuous improvement.

Primary framing: agent lifecycle


1
Brief
Turn a goal into a structured agent brief



2
Design
Generate and inspect architecture candidates



3
Compare
Review tradeoffs, debate, and rationale



4
Evaluate
See benchmark, scenario, and safety results



5
Approve
Resolve risks before release



6
Deploy
Canary, observe, and roll back safely



7
Improve
Learn from runs, failures, and outcomes

Who this serves


Agent builders
Guided creation, architecture confidence, and clear next steps.


ML / platform
Evaluation evidence, runtime visibility, and version control.


Governance
Approvals, policy, auditability, and release-safe visibility.


Executives
Portfolio outcomes, risk posture, and adoption signals.

Design principle
Every key screen should answer six questions: what this agent does, how it was designed, why the platform chose it, whether it is safe, what changed, and whether it is ready to deploy.

Product stance
Do not organize the user experience around backend nouns like Supervisor or Swarm. Those belong in diagnostics and audit views. The main UX should feel like an agent studio with clear lifecycle stages.

Why a three-panel layout
Left = progress and orientation. Center = main artifact. Right = advice, risk, and next-step guidance. This is the most consistent pattern across the product.

Decision support

Three-layer review
1) summary verdict, 2) grouped failures, 3) raw evidence. This keeps the screen useful for both executives and engineers.

Release-safe behavior
Never force a user to parse raw traces before understanding the headline. The recommendation block should state approve, revise, or block.

Design detail
Use color sparingly and semantically: green for verified, amber for caution, red for blocked or degraded.
