Agent Architect Pro — API & Interface Contract Pack
Detailed contract pack for the recommended production architecture: request/response schemas, event contracts, identity, error handling, and compatibility rules.

1. Purpose and scope
This document defines the interface contracts required to implement Agent Architect Pro as a governed multi-agent platform. It covers synchronous APIs, asynchronous events, identity and security envelopes, versioning, validation rules, idempotency expectations, and baseline service-level assumptions.
• The platform must support requirement intake, planning, retrieval, evaluation, deployment, observability, and continuous improvement.
• The primary control-plane roles are Supervisor, Planner, Release Manager, and Identity/Policy services.
• The primary execution-plane roles are Swarm Orchestrator, Runtime workers, Tool Broker, Model Gateway, and Result Aggregation.

5.1 Supervisor — POST /v1/runs


• Required validations: objective present, tenant_id present, cost/deadline inside tenant policy limits.
• Idempotency: repeated calls with the same client_request_id within 24h must return the same run_id unless the request payload differs.
• State machine: planning → queued → running → awaiting_review → completed | failed | cancelled.

5.4 Tool Broker — POST /v1/tools/execute

• Policy check precedes execution. Missing grant or expired token returns AUTHZ_DENIED.
• All successful and blocked calls generate AuditEvents.
• Long-running tools should return an operation_id and support callback or polling semantics.
5.5 Model Gateway — POST /v1/models/generate

• The gateway hides provider/model selection. Clients specify policy and task_type, not vendor-specific parameters.
• Responses include token counts, latency_ms, model_route, and cache_hit metadata.
• Streaming mode is optional; the same contract supports both synchronous and chunked response delivery.

9. Acceptance checklist
• Every service must publish an OpenAPI spec or protobuf equivalent derived from these contracts.
• Every privileged action must produce a matching AuditEvent with trace_id and actor identity.
• Every client SDK must expose idempotency, correlation IDs, and standard error parsing.
• Contract tests must validate both happy-path and policy-failure behavior for each critical endpoint.
• UI and workflow surfaces should consume only documented fields and must ignore additive fields they do not recognize.

--- TABLE 2 ---
Rule | Contract decision
API style | JSON over HTTPS for public/service APIs. Internal high-throughput service-to-service calls may use gRPC behind the same logical contract.
Versioning | URI major versioning: /v1/… . Non-breaking additive changes are allowed within v1. Breaking changes require /v2.
Authentication | All privileged calls require a bearer token. Runtime workers use short-lived workload tokens issued by the Token Service.
Idempotency | POST operations that create or trigger state transitions must accept Idempotency-Key or client_request_id.
Tracing | Every request and event must carry trace_id, run_id when available, and task_attempt_id when execution-related.
Error model | All services return a standard error envelope with code, message, retryability, correlation id, and optional field violations.

--- TABLE 18 ---
Area | Required rule
Transport | TLS everywhere. Internal service mesh mTLS recommended for service-to-service calls.
Tokens | Workload tokens should be short-lived, audience-bound, and least-privilege.
PII handling | Sensitive fields must be tagged and excluded from broad observability payloads.
Schema evolution | Additive fields only within v1. Consumers must ignore unknown fields.
Deprecation | Deprecated fields require notice in contract docs and a minimum two-release compatibility window.
Rate limits | Enforce per-tenant and per-service quotas with explicit 429 handling and retry-after guidance.
