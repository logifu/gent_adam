Agent Architect Pro — Figma Page-by-Page Build Brief
Designer-facing build document aligned to the approved UX spec, screen inventory, and Figma build-order workbook.

1. Build philosophy
Design the product around the lifecycle of an agent: Brief → Design → Compare → Evaluate → Approve → Deploy → Monitor → Improve. That means the Figma file should privilege role clarity, approval moments, and progressive disclosure over backend complexity.

3. Milestones and review gates

Foundations and shared system
This phase establishes the non-negotiable shared system for the whole file. Do not let individual pages invent their own shell, status language, or spacing rules.



Global and entry experience
This phase defines first impression, orientation, and how new work begins. The product must feel premium, structured, and calm before it feels powerful.



Studio core workflow
This phase contains the most strategic product surfaces. It must make architecture choices, tool setup, memory setup, evaluation, and release readiness feel understandable and defensible.



Studio extended and review surfaces
This phase adds supporting context and evidence. These pages deepen trust and technical review, but they should not outshine the main builder flow.



Runtime, runs, deployments, and debugging
This phase turns the system into an operational product. It must connect releases, runtime behavior, debugging, and rollout decisions without losing readability.



Governance and knowledge views
This phase formalizes trust, policy, evidence, and retrieval assets. Structure and actionability matter more here than visual flourish.



Executive and admin surfaces
This final phase serves secondary stakeholders. Build it after the core creation-to-runtime journey is stable and polished.



4. Prototype flow build plan
Wire only the flows that prove the product narrative early. A convincing first prototype should demonstrate creation, evaluation, release, deployment monitoring, and runtime investigation.
5. Designer QA and handoff checklist

--- TABLE 21 ---
Checkpoint | What to confirm before handoff
Foundations | Spacing, type, tokens, status semantics, shell behavior, and core components match across pages.
Screen readiness | Each P0 screen has a clean default state, at least one meaningful edge state, and named frames that match the workbook.
Prototype quality | P0 flows are connected, interactive hotspots are obvious, and transitions preserve user context.
Trust signals | Risk, evidence, approvals, and state changes are visible in summary view before deeper inspection.
Data realism | Use believable sample content, not lorem ipsum, for agent names, failures, evaluations, and releases.
Review package | Frame names, section names, and comments are ready for design review and engineering translation.
