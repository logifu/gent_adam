Agent Architect Pro — Component-by-Component Figma Library Brief
Professional build brief for the design-system library that supports Studio, Evaluation, Deployment, Observability, and Governance workflows.

1. Objective and design stance
This brief defines the Figma library required to design Agent Architect Pro as a trustworthy enterprise product. The goal is not to create a visually impressive component dump. The goal is to build a systemized library that helps the team ship coherent screens for complex AI workflows, make governance visible, and reduce friction between product design, engineering, and future prototyping.
• Design the library around product workflows, not backend service names.
• Keep primitives universal, but create domain components where this platform truly needs them.
• Bias toward components that support comparison, explainability, review, and approval.
• Use a layered model: foundations → primitives → composites → domain modules → page patterns.
• Treat accessibility, states, and handoff metadata as part of the component definition, not a later cleanup step.

3. Naming, structure, and variant rules
• Use a consistent slash hierarchy: Category / Component / Variant / State. Example: Navigation / SideNav / Item / Active.
• Separate visual size from semantic type. Example: Badge / Status / Medium and Badge / Metric / Medium should not be a single overloaded set.
• Prefer boolean or enumerated properties over duplicate components. Typical properties: size, emphasis, state, icon-left, icon-right, density, has-helper, has-prefix, has-suffix.
• Treat content slots explicitly in documentation: title, subtitle, metadata row, action row, footer, evidence chip group, right rail actions.
• Use component descriptions in Figma to specify accessibility expectations, content limits, and intended use cases.
4. Foundation layer

5. Component library map
The following categories are the recommended minimum component inventory for v1. They are organized from universal primitives to domain-specific modules.
5.1 App shell and navigation
These components give the product a stable frame. If this category is inconsistent, the whole platform will feel fragmented.

5.2 Core actions and feedback
These are standard UI primitives, but this product depends on strong state design because users make high-consequence decisions.

5.3 Forms and input controls
The platform needs guided intake and structured configuration, so forms must feel deliberate rather than generic.

5.4 Content containers and data display
This category supports dashboards, review screens, and dense information layouts.

5.5 AI-native design and reasoning modules
This is the category that differentiates Agent Architect Pro from a generic dashboard or cloud admin tool.

5.6 Evaluation, release, and governance modules
These modules support the trust layer of the product. They should feel formal, clear, and defensible.

5.7 Deployment and runtime modules
These modules let the product shift from design studio into operating system mode without changing visual language.

5.8 Overlay and utility components
These support usability, but they must not become the main way users understand the product.

6. Pattern layer and screen recipes

7. Build sequence for the Figma library
• Phase 1 — Foundations: variables, typography, spacing, elevation, grid, icons, and semantic colors.
• Phase 2 — Primitives: buttons, inputs, chips, badges, tabs, toggles, alert patterns, loaders, and tables.
• Phase 3 — App shell: side nav, top bar, page header, command bar, filter bar, and right context panel.
• Phase 4 — Domain modules: architecture card, comparison matrix, debate panel, evidence drawer, evaluation verdict, release diff, approval checklist.
• Phase 5 — Patterns and screen recipes: Studio layout, compare workspace, review stack, ops dashboard, and executive dashboard compositions.
• Phase 6 — Prototype blocks and QA: interactions, edge-case states, accessibility checks, and handoff annotations.

8. Component documentation standard

9. Quality gate for publishing library components
• Every component must have idle, hover, focus, pressed, disabled, and loading/error states where applicable.
• Every state change must preserve layout stability unless the pattern intentionally expands or collapses.
• Components used in governance and deployment flows must support explicit status language and not rely on color alone.
• Table, timeline, and audit components must be legible in compact density, because Ops use cases demand high information density.
• Domain modules must include rationale and evidence affordances if they represent an AI-generated recommendation or verdict.
• Before publishing, verify component naming, property cleanliness, auto layout, resizing logic, and consistent token usage.
10. Final recommendation
The strongest library strategy for Agent Architect Pro is a disciplined shared system with a small number of genuinely product-specific modules. The product should not be built entirely from generic SaaS components, because comparison, evidence, evaluation, approval, and runtime trust are central to the experience. At the same time, it should not become an undisciplined collection of one-off cards. A good Figma library here is opinionated, sparse where the product can stay standard, and highly intentional where AI governance and lifecycle workflows need custom interaction patterns.
