Agent Architect Pro — Master Execution Command Manual
Comprehensive command-and-instruction document for turning the architecture, UX, API contracts, and roadmap into executable team action.

1. What this document is for
This manual translates the project from reference artifacts into command-style execution. It is meant to remove ambiguity, shorten decision cycles, and prevent rework by making the order of operations explicit.
• Treat the product as a lifecycle studio: Brief → Design → Compare → Evaluate → Approve → Deploy → Monitor → Improve.
• Build the platform as six planes with clear interfaces: ingress/control, execution, knowledge, trust, improvement, and operations.
• Freeze contracts early, keep the online path reliable, and gate every production step with evidence, approval, and observability.
• Do not allow frontend, evaluation, or release work to outrun backend contracts and identity/audit foundations.

Non-negotiable execution rule
If a task creates future ambiguity, replace it with a more explicit task. This project is too large for vague backlog items such as “build architecture” or “do observability later.” Every command should name owner, input, output, dependency, and exit criteria.

--- TABLE 11 ---
Gate | Stop if this is missing
Architecture gate | Contracts are undefined, plane ownership is disputed, or backend and frontend are designing against different mental models.
Alpha gate | No single auditable vertical slice exists from run intake to final response.
Beta gate | Product workflow works only with mocks, release controls are absent, or policy bypasses remain.
Production-candidate gate | No tested canary/rollback path exists, no alerts are live, or no incident response playbook exists.
Launch gate | Pilot users cannot complete the lifecycle through the product UI, or support/observability ownership is unclear.

How to use this manual weekly
At the start of each sprint, convert every planned story into a command with owner, input, explicit output, dependency, and exit criteria. At the end of the sprint, review only against exit criteria — not against effort spent.
