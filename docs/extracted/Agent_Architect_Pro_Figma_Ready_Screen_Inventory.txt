
=== SHEET: README ===
Agent Architect Pro — Figma-Ready Screen Inventory | 
Structured inventory for Figma file setup, screen planning, reusable components, flows, and foundations. | 
 | 
Purpose | 
Use this workbook as the source-of-truth inventory before building high-fidelity design files. It is optimized for a Figma workflow: clear screen IDs, route structure, states, reusable components, flow mappings, and a starter token system aligned to the product strategy. | 
 | 
Recommended Figma file structure | 
00_Cover | Project title, version, owners, status, links to research/specs.
01_Foundations | Color tokens, type scale, spacing, radius, elevation, iconography, grid rules.
02_Global Components | App shell, navigation, chips, cards, tables, forms, drawers, modals, timeline, evidence modules.
03_Core Flows | Low-fi and mid-fi flow frames for Create Agent, Evaluate, Deploy, Monitor, Govern.
04_Screens | Production screen sets organized by product area and screen ID.
05_Prototype | Primary clickable flow for stakeholder review.
06_Archive | Deprecated concepts, prior exploration, design notes.
 | 
Naming and handoff rules | 
• Use the exact Screen ID from the Screen_Inventory tab in each Figma frame title. | 
• Name frames as: [Screen ID] – [Screen Name] – [State], e.g., STU-004 – Architecture Comparison – Default. | 
• Keep one canonical component set; do not duplicate local variants unless the behavior genuinely differs. | 
• Design light and dark mode only after the default light-mode flow is approved. | 
• Every screen needs at least these states defined: default, loading, empty, error, and permission-limited where relevant. | 
• For enterprise handoff, connect components to usage notes and data requirements, not just visuals. | 
 | 
 | 
Priority legend | 
P0 | Must design in first Figma sprint
P1 | Needed soon after core flow approval
P2 | Secondary / later release
P3 | Future / optional
