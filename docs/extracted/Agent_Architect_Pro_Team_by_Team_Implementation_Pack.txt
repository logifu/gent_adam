Agent Architect Pro — Team-by-Team Implementation Pack
Execution pack for Backend, Frontend, ML/AI, DevOps/SRE, and Governance/Security teams.

1. How to use this pack
• Use this document as the working operating brief for team leads and individual contributors.
• Each team section defines ownership, deliverables, dependencies, cadence, and definition of done.
• Treat this as the bridge between architecture documents and day-to-day execution.
• Do not start work only from ideas; start from the assigned team charter and its first-30-days checklist.

2. Cross-team operating model

3. Shared rules for all teams
• Every team must work from versioned contracts, not informal assumptions.
• Every feature must emit usable telemetry: logs, traces, metrics, and contextual identifiers.
• No team may introduce undocumented fields or hidden workflow branches into production paths.
• Security and governance are not a final hardening step; they are part of feature completion.
• Release readiness requires code, tests, docs, telemetry, and rollback understanding.
4. Backend / Platform Team
Mission: build and stabilize the core services that make the product operationally real — API gateway integration, supervisor, planner, swarm orchestration, runtime lifecycle, tool broker, model gateway, and release manager support.

5. Frontend / Product Experience Team
Mission: turn the platform into an understandable product using the lifecycle-first experience model — Home, Studio, Evaluation, Release, Runtime, Governance, and Executive surfaces.

6. ML / AI Team
Mission: own the intelligence plane — retrieval quality, model routing policy, candidate generation, evaluation harnesses, replay/simulation, and evidence packaging that makes the system trustworthy.

7. DevOps / SRE Team
Mission: make the platform deployable, observable, recoverable, and safe to operate through environments, pipelines, secrets, scaling, and incident response readiness.

8. Governance / Security Team
Mission: ensure identity, policy, approval, audit, and operational trust are built into the platform from the beginning and not treated as late-stage paperwork.

9. Team handoff matrix

10. Team-level review cadence
• Daily: team standups focused on blockers, dependencies, and contract drift.
• Twice weekly: cross-team integration sync involving Backend, Frontend, ML, DevOps, and Governance leads.
• Weekly: architecture and product review focused on the current vertical slice.
• Per sprint: demo, risk review, acceptance check, and next-sprint dependency confirmation.

11. Team-by-team definition of done summary

12. Final recommendation
Use this implementation pack as the team-level command layer beneath the execution manuals. The manuals explain the project and the phase-by-phase sequence; this document tells each team exactly how to participate without ambiguity. If the project keeps these ownership boundaries and handoff rules intact, it will move faster and with far less rework.
